OpenKore CVS Guide

Introduction to CVS

One of the problems of multiple developers working on one file/project is that they have to merge their changes with each other.

CVS (Concurrent Versions System) is a tool that makes it easier for multiple developers to work on one project. Basically, it helps you with:

  • Merge your changes to other people's changes.
  • Revert back changes that you've made.
CVS is widely used in the open source world, and is a very useful tool.

Warning

CVS is a developer tool. The code in CVS is "bleeding-edge": some code may be experimental and not well tested. The code may not work, it may contain lots of bugs, it may format your hard drive. The code in CVS is not meant to be used by the general public, but is meant as a place for developers to store the latest versions of their source code.

Basic concept

The latest public release of OpenKore does not necessarily contain the latest code (the code that developers are working on at this moment). Instead, that code is stored on the SourceForge CVS server (a CVS repository).
You shouldn't just take the latest public release and modify that. Instead, you should get the code from CVS and work on that instead.

The OpenKore CVS repository contains several modules. A module is a subfolder containing a part of OpenKore. These are the available modules at this moment:
Module name Description
openkore The main openkore source code, including the code for Tools.dll and Inject.dll. This is probably what you're looking for.
confpack The default configuration files (control/*).
tablepack The default table files (tables/*).
fieldpack *.fld files.
plugins Simple example plugins for OpenKore.
extract Tools used to automatically generate recvpackets.txt
grftool Source code of GRF Tool, libgrf and related stuff.

Basically, this is what you have to do if you want to work on OpenKore:

  1. Download (checkout) source code from CVS.
  2. Modify the source code.
  3. Merge your changes with the latest changes in CVS.
    • If you're a developer, upload (commit) your changes to CVS.
    • If you're not a developer, make a patch and submit it to the forums.
Each step will be explained.

Note: this guide assumes that you are using MS Windows. If you're a Linux developer, you probably already know how to use CVS.

Preparations

First, you will need to install a CVS client in order to access CVS servers, just like you need a web browser to access HTTP servers. There are many CVS clients for Windows, but I recommend TortoiseCVS because of it's simplicity and integration with Windows Explorer: http://www.tortoisecvs.org/

This guide assumes that you're using TortoiseCVS.

Tip: once you've installed TortoiseCVS, you should tell it to use compression when uploading/downloading stuff. Rightclick on an empty spot in any folder and choose "CVS->Preferences...". Go to the "Advanced" tab and choose "3 - Low" for Network compression. Do NOT choose a higher level of compression! Anything higher than 3 won't give much better results, but will stress the server more. SourceForge already has to deal with lots of traffic, so please don't set compression higher than 3.
